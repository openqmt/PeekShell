# PeekShell 开发规范

本文约定 PeekShell（Tauri 2 + Rust + Vue 3 / TypeScript / Vite）的编码与协作标准。提交与评审均以此为准。

---

## 1. 核心原则

1. **只写必要的代码**：不为「以后可能用到」预留无调用的变量、函数、类型或文件。
2. **禁止冗余**：同一逻辑只保留一处；不用的绑定、参数、分支、注释掉的旧代码一律删除。
3. **注释说明意图**：注释解释「为什么」和「约束」，而不是复述代码字面意思。
4. **改动范围要小**：只改与当前任务相关的代码；不做无关重构与顺手格式化整文件。
5. **与现有风格一致**：命名、目录、错误处理、组件写法跟随仓库已有模式。

---

## 2. 禁止冗余变量与函数

### 2.1 变量

- 禁止声明后从未读取的变量、常量、解构字段。
- 禁止仅为「中转」而存在的一层赋值（能直接用表达式就不要多一个名字），除非命名能显著提高可读性。
- 禁止 `let` 赋值后从未重新赋值的变量（应使用 `const` / Rust 的不可变绑定）。
- 禁止无意义的布尔标志叠加；能用早返回或枚举表达时不要堆 `if (flag1 && flag2)`。

```ts
// 不好：中间变量从未真正增加可读性，且易成死变量
const raw = props.host.ip
const ip = raw
connect(ip)

// 更好
connect(props.host.ip)
```

### 2.2 函数与方法

- 禁止未被调用的函数、方法、组件、composable、Rust `fn`。
- 禁止「只包一层」的透传函数（输入原样交给另一个函数且无校验、无日志、无适配）。
- 禁止复制粘贴出两份几乎相同的函数；抽公共逻辑，或保留一处并参数化。
- 删除功能时，同步删除仅被该功能使用的辅助函数与类型。

### 2.3 前后端约定

| 位置 | 要求 |
|------|------|
| Vue / TS | 未使用的 `import`、props、emit、Pinia state/action 必须删除 |
| Rust | 未使用的 `use`、结构体字段、`#[tauri::command]`（若前端已无调用）必须删除 |
| 共享类型 | `packages/shared` 中只保留前后端真实使用的 schema |

## 3. 注释规范

### 3.1 必须写注释的地方

- **模块 / 文件头**：该文件职责、所属分层（SSH / Agent / UI / 存储）。
- **导出的公共 API**：Tauri command、Pinia store 的对外 action、可复用 composable。
- **非显而易见的业务规则**：例如命令风险分级、为何不能自动执行、凭证为何走钥匙串。
- **安全与并发相关**：SSH 会话生命周期、审批后才执行、脱敏规则、锁与异步边界。
- **变通与临时方案**：注明原因与可移除条件（避免永久「临时」）。

### 3.2 注释写什么

```ts
// 好：说明约束与原因
// 必须先经用户确认：AI 提议的命令不得直接写入 PTY
async function executeApproved(commandId: string) { ... }

// 不好：复述代码
// 执行命令
async function executeApproved(commandId: string) { ... }
```

```rust
/// 采集远端主机指标，供侧栏展示与 AI 上下文。
/// 失败时返回上一次成功快照，避免 UI 闪断。
pub async fn collect_host_metrics(session_id: &str) -> Result<HostMetrics> { ... }
```

### 3.3 注释不要写什么

- 不要写能从类型和命名直接看懂的内容。
- 不要用注释保留大段已删除代码（用 Git 历史）。
- 不要留下矛盾或过期注释；改代码时同步改注释。
- 中英文可混用，但同一文件内风格统一；对外文档优先中文。

### 3.4 推荐格式

- TypeScript / Vue：`//` 行注释；复杂逻辑可用简短块注释。
- Rust：公共项用 `///`文档注释；模块内用 `//`。
- 组件：对非常规交互（折叠侧栏、确认卡片状态机）在 `<script>` 顶部用几行说明数据流。

## 4. 按技术栈的补充约定

### 4.1 Vue 3 + TypeScript

- 优先 Composition API + `<script setup lang="ts">`。
- 组件单一职责；主机列表、连接弹窗、终端、AI 面板拆开，避免巨型单文件。
- 状态：跨页/跨面板用 Pinia；仅本地 UI 用 `ref` / `reactive`。
- 模板中避免复杂表达式；抽到 computed 或函数，并保证该函数确有复用或可读性价值。

### 4.2 Rust / Tauri

- SSH、凭证、命令执行放在 Rust；前端只负责展示与用户确认。
- command 入参 / 出参类型明确，避免松散的 `Value` / `any` 来回传。
- 错误用可区分的错误类型或错误码返回前端，便于提示用户。

### 4.3 AI Agent

- 提议与执行必须分离；执行入口只接受「已批准」的 command id。
- 风险分级、脱敏、审计相关逻辑集中在明确模块，并写清注释。

---

## 5. 命名与结构

- 命名表达用途：`hostList`、`collectHostMetrics`、`executeApprovedCommand`。
- 布尔名用 `is` / `has` / `can` 前缀：`isCollapsed`、`canExecute`。
- 文件名与导出职责一致；不要 `utils.ts` 无限膨胀——按领域拆分（如 `ssh/`、`agent/`、`hosts/`）。

---

## 6. 提交前自检清单

提交或发起 PR 前确认：

- [ ] 没有未使用的变量、函数、import、死代码
- [ ] 没有只包一层的无意义封装
- [ ] 公共 API、安全与业务约束处有必要注释
- [ ] 注释与代码一致，无过期说明
- [ ] 未引入与任务无关的改动
- [ ] 能本地编译 / 类型检查通过（`vue-tsc` / `cargo check` 等）
